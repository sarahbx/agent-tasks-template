# REQ-9: Post-Quantum Cryptography Readiness

## Requirement

**All projects processed through the SDLC pipeline that involve TLS configuration, certificate management, key exchange, digital signatures, or cryptographic operations must meet the following cryptographic posture requirements:**

1. **TLS 1.3 is the minimum acceptable TLS protocol version** for all projects (greenfield and existing).
2. **Greenfield projects must use PQC-ready hybrid key exchange** from the start, defaulting to the highest available security level (ML-KEM-1024 / NIST Level 5).
3. **Existing projects must receive PQC migration recommendations**, but breaking changes require explicit human approval before implementation.
4. **All algorithm and cipher suite recommendations must be verified** against current publications from five independent authoritative sources before enforcement.
5. **Recommendations must be reviewed quarterly** at minimum, with earlier review triggered by any authoritative source guidance change.

This requirement specifies **algorithm families** (e.g., ML-KEM, ML-DSA, SLH-DSA, AES-256-GCM, ECDHE), not specific cipher suite strings. Engineers select platform-appropriate cipher suite configurations within the required algorithm families.

## Rationale

```
Why PQC readiness as a requirement?
----------------------------------------------------------------------
Quantum threat:     Cryptographically relevant quantum computers
                    (CRQCs) threaten all classical asymmetric
                    cryptography (RSA, ECDSA, ECDH, DH). The
                    timeline is uncertain, but the "harvest now,
                    decrypt later" threat is active today.

Standards maturity: NIST has finalized FIPS 203 (ML-KEM), FIPS 204
                    (ML-DSA), and FIPS 205 (SLH-DSA). NSA CNSA 2.0
                    mandates PQC adoption timelines. CISA recommends
                    immediate migration planning. IETF is
                    standardizing PQC integration into TLS. ISO/IEC
                    JTC 1/SC 27 is incorporating PQC into
                    international standards.

Moving target:      Cryptographic best practices evolve as standards
                    mature, algorithms are analyzed, and threats
                    change. A static recommendation becomes stale.
                    The five-source verification and quarterly review
                    mechanism ensures recommendations remain current.

Posture floor:      Without an enforceable minimum, projects may use
                    deprecated TLS versions or classical-only
                    cryptography indefinitely, accumulating
                    cryptographic debt.
----------------------------------------------------------------------
```

## Authoritative Sources

All algorithm and cipher suite recommendations under this requirement must be verified against current publications from the following five independent authoritative sources:

```
Source Verification Requirements
----------------------------------------------------------------------
1. NIST (nist.gov)
   Scope: FIPS standards, SP 800-series, PQC standardization
   Weight: Authoritative for algorithm standardization

2. NSA (nsa.gov)
   Scope: CNSA 2.0 suite, cryptographic guidance
   Weight: Authoritative for security level requirements

3. CISA (cisa.gov)
   Scope: PQC migration guidance, cybersecurity advisories
   Weight: Authoritative for implementation guidance

4. IETF (ietf.org)
   Scope: RFCs for TLS, key exchange, PQC protocol integration
   Weight: Authoritative for protocol-level standards
   IMPORTANT: Only ratified RFCs carry authoritative weight.
   Internet-Drafts (I-Ds) are informational context only and
   do not constitute ratified recommendations. When a draft is
   the sole IETF reference for a PQC mechanism, note this as a
   maturity gap and require the remaining four sources to provide
   sufficient coverage. [SAR: SEC-005]

5. ISO/IEC JTC 1/SC 27 (iso.org)
   Scope: International cryptographic standards (ISO/IEC 18033,
          14888 series, PQC-related work items)
   Weight: Authoritative for international regulatory alignment
----------------------------------------------------------------------

Verification process:
  - Confirm each source's publication is current (not superseded)
  - Record the publication date for each source consulted
  - Confirm recommendation alignment across all five sources
  - If ANY source has deprecated, retracted, or issued an advisory
    against a recommended algorithm: the recommendation is BLOCKED
    until the conflict is resolved

Source unavailability fallback:
  When a live authoritative source is temporarily unavailable,
  verification may use the most recent cached/archived version,
  provided ALL of the following conditions are met:
    (a) The cached version is less than 90 days old
    (b) No known retraction or update has been issued
    (c) The unavailability and use of cached source is documented
        in the gate artifact
    (d) The human explicitly approves proceeding with cached
        source verification before the gate advances
  This is NOT an automatic fallback. The agent must present the
  cached source date and request human confirmation. [SAR: SEC-003]
----------------------------------------------------------------------
```

## Greenfield Project Requirements

Greenfield projects must start with an enhanced cryptographic posture:

```
Greenfield Cryptographic Posture
----------------------------------------------------------------------
TLS version:        TLS 1.3 mandatory. No exceptions except the
                    narrow legacy compatibility exception below.

Key exchange:       PQC-ready hybrid key exchange required.
                    Algorithm family: ML-KEM (FIPS 203).
                    Default security level: ML-KEM-1024 (NIST
                    Level 5). Lower levels (ML-KEM-768, ML-KEM-512)
                    are acceptable ONLY when the engineer demonstrates
                    meaningful, documented performance issues that
                    cannot be resolved at Level 5. The burden of
                    proof is on downgrading, not on upgrading.

Digital signatures: ML-DSA (FIPS 204) or SLH-DSA (FIPS 205)
                    recommended for new certificate and signing
                    infrastructure. Classical signatures (ECDSA,
                    Ed25519) remain acceptable during the transition
                    period but PQC signature adoption should be
                    planned.

Symmetric crypto:   AES-256-GCM or ChaCha20-Poly1305 for symmetric
                    encryption. AES-128 is acceptable but AES-256 is
                    preferred for quantum margin.

Hybrid fallback:    Silent fallback from hybrid (classical + PQC) to
                    classical-only key exchange is PROHIBITED. If the
                    PQC component of a hybrid handshake fails, the
                    connection must either:
                      (a) Fail closed (refuse the connection) — this
                          is the RECOMMENDED default
                      (b) Succeed but generate a logged, operator-
                          visible alert
                    The choice between (a) and (b) must be documented
                    in the project ADR and approved by the human.
                    [SAR: SEC-006]

Legacy exception:   TLS versions below 1.3 are permitted ONLY for a
                    compatibility layer that must interoperate with
                    systems that absolutely cannot be upgraded. This
                    exception requires:
                      (a) Documentation in the project ADR
                      (b) Justification that the legacy system cannot
                          be upgraded
                      (c) Explicit human approval
                      (d) The compatibility layer must not weaken the
                          security posture of any non-legacy component
                      (e) The legacy layer must use the strongest
                          available TLS 1.2 cipher suites (AES-256-GCM
                          with ECDHE key exchange family at minimum)
                      (f) TLS 1.1 and TLS 1.0 are prohibited even in
                          legacy contexts unless absolutely unavoidable,
                          with separate human approval required
                    NOTE: Enforcing minimum cipher suites on a legacy
                    layer may be a breaking change. The engineer must
                    flag this and obtain human approval. [SAR: SEC-002]
----------------------------------------------------------------------
```

## Existing Project Requirements

Existing (brownfield) projects must meet the TLS 1.3 minimum and receive PQC migration guidance:

```
Existing Project Cryptographic Posture
----------------------------------------------------------------------
TLS version:        TLS 1.3 is the minimum. Projects on TLS 1.2
                    must upgrade. If the upgrade would break existing
                    functionality, the engineer must:
                      (a) Document the breaking changes
                      (b) Present a migration plan
                      (c) Obtain explicit human approval before
                          implementing

PQC adoption:       PQC hybrid key exchange is RECOMMENDED but not
                    required for existing projects. The recommendation
                    must be presented in the ADR with:
                      (a) Current cryptographic posture assessment
                      (b) Specific PQC migration steps
                      (c) Impact analysis on existing functionality
                      (d) Timeline recommendation aligned with CNSA
                          2.0 milestones
                    If the human approves PQC adoption, it follows
                    the same standards as greenfield (ML-KEM-1024
                    default, hybrid fallback rules, etc.)

No silent           No change to an existing project's cryptographic
downgrade:          configuration may reduce its security posture.
                    This reinforces REQ-5 specifically for
                    cryptographic transitions. Downgrade scenarios
                    include but are not limited to:
                      - Removing PQC support that was previously
                        enabled
                      - Reducing TLS version
                      - Adding weaker cipher suites to the
                        configuration
                      - Disabling certificate validation
----------------------------------------------------------------------
```

## Legacy Compatibility Layer Scoping

When a legacy compatibility exception is granted, it must be narrowly scoped:

```
Legacy Exception Scoping Rules [SAR: SEC-001]
----------------------------------------------------------------------
Enumeration:        The exception applies ONLY to explicitly
                    enumerated endpoints or connection targets. Broad
                    categories (e.g., "all external partners") are
                    not acceptable. Each legacy endpoint must be
                    individually listed.

Sunset dates:       Each exception must include a sunset date or a
                    review trigger. The sunset date should align with
                    the legacy system's planned upgrade or
                    decommission timeline. If no upgrade is planned,
                    a review date no more than 12 months out must be
                    set.

Network isolation:  Legacy-TLS traffic must not be routed to or from
                    non-legacy components without traversing TLS 1.3.
                    The legacy compatibility layer must be isolated so
                    that its reduced security posture cannot affect
                    components operating at the full TLS 1.3 + PQC
                    posture.
----------------------------------------------------------------------
```

## Review Cadence

```
Recommendation Review Schedule
----------------------------------------------------------------------
Quarterly review:   At minimum, cipher suite and algorithm family
                    recommendations must be reviewed against all five
                    authoritative sources every quarter (calendar
                    quarter).

Triggered review:   Earlier review is REQUIRED if any authoritative
                    source issues:
                      - Updated guidance affecting recommended
                        algorithms
                      - Deprecation notices for any recommended
                        algorithm
                      - Security advisories affecting recommended
                        algorithms or protocol versions

Ad-hoc review:      Any agent or human may trigger a review at any
                    time if they become aware of information that
                    may affect the current recommendations.

Review output:      Each review must produce a dated record
                    confirming:
                      - All five sources were consulted
                      - Current recommendations remain valid, OR
                      - Specific recommendations require update
                        (with proposed replacements)
                    Changes to recommendations require human approval
                    before taking effect.
----------------------------------------------------------------------
```

## Scope

- **Applies to:** All projects processed through the SDLC pipeline that involve TLS configuration, certificate management, key exchange, digital signatures, or cryptographic operations
- **Greenfield projects:** Enhanced posture (TLS 1.3 mandatory, PQC hybrid required, ML-KEM-1024 default)
- **Existing projects:** TLS 1.3 minimum, PQC recommended with human-gated migration path
- **Not applicable:** Projects with no TLS, cryptographic, or certificate-related components

## Enforcement Rules

```
Gate 1 (Architect):    ADR must identify all TLS and cryptographic
                       components in the design. Must classify the
                       project as greenfield or existing. Must flag
                       any legacy compatibility requirements. Must
                       verify algorithm recommendations against all
                       five authoritative sources and record
                       publication dates. Any design using TLS below
                       1.3 without a documented legacy exception is
                       a REQUIRED revision.

Gate 2 (Security       SAR must verify cipher suite and algorithm
Architect):            recommendations against all five authoritative
                       sources. Must verify greenfield/brownfield
                       classification and risk. Must verify no
                       downgrade scenarios exist. Must verify hybrid
                       fallback behavior is specified. Any single-
                       source recommendation is a HIGH finding. Any
                       silent hybrid fallback is a HIGH finding.

Gate 3 (Team Lead):    Sprint Brief must surface cryptographic posture
                       decisions, legacy exceptions, and any pending
                       human approvals for breaking changes.

Gate 4 (Engineer):     Implementation must enforce TLS 1.3 minimum.
                       Greenfield: PQC hybrid with ML-KEM-1024
                       default and specified fallback behavior.
                       Legacy exceptions must follow scoping rules.
                       All SAR mitigations must be implemented.

Gate 5 (Code           Any TLS configuration below 1.3 without a
Reviewer):             documented legacy exception is a REQUIRED
                       change. Any silent hybrid fallback is a
                       REQUIRED change. Any algorithm recommendation
                       not verified against all five sources is a
                       REQUIRED change.

Gate 6 (Quality        Same enforcement as Gate 5. Additionally
Engineer):             verify that algorithm recommendations are
                       current (check source publication dates
                       against the current date and the quarterly
                       review cadence).

Gate 7 (Security       Cryptographic posture violations are CRITICAL
Auditor):              findings. Verify: TLS version, PQC hybrid
                       configuration, algorithm source verification
                       dates, legacy exception scoping, hybrid
                       fallback behavior, and quarterly review
                       compliance. Any unverified algorithm
                       recommendation is a CRITICAL finding.
```
