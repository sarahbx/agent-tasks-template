# REQ-5: Security Posture Preservation

## Requirement

**No interaction with any service, system, tool, or external resource — in any form — may reduce the security posture of data passing through the pipeline.** This applies to every gate, every agent, and every skill invocation. A security posture reduction is any action that weakens confidentiality, integrity, availability, authentication, authorization, or auditability of data or processes compared to their state before the interaction.

## Rationale

```
Why security posture preservation?
──────────────────────────────────────────────────────────────────────
Data integrity:   Data that enters the pipeline with a given security
                  posture must exit with at least that posture. No
                  intermediate step may downgrade protections.

Trust chain:      Each gate in the pipeline trusts the output of prior
                  gates. If any step silently reduces security posture,
                  downstream gates operate on false assumptions.

Defense in        This requirement creates a universal floor. Individual
depth:            gates may add controls; none may remove controls that
                  were present at the start of the pipeline.

Cross-domain:     This applies regardless of what service or system is
                  being interacted with — Jira, git, CI/CD, external
                  APIs, local file systems, or any future integration.
──────────────────────────────────────────────────────────────────────
```

## Scope

- **Applies to:** All agents, all gates, all skill invocations, all interactions with external systems
- **Examples of violations:**
  - Sending data over an unencrypted channel when an encrypted channel was available
  - Storing credentials in plaintext when the source used a secrets manager
  - Logging sensitive data that was previously redacted
  - Disabling a security control to work around an integration issue
  - Transmitting data to a less-trusted system without explicit human approval
- **Not a violation:** Using a system's native security controls even if they differ from the pipeline's (e.g., Jira's access controls govern Jira tickets)

## Enforcement Rules

```
Gate 1 (Architect):    ADR must identify every external system interaction
                       and confirm the security posture of data is preserved
                       or improved at each boundary crossing. Any design
                       that reduces posture is a REQUIRED revision.

Gate 2 (Security       SAR must verify that every data flow crossing a
Architect):            trust boundary preserves security posture. Any
                       posture reduction is at minimum a HIGH finding.

Gate 3 (Team Lead):    Sprint Brief must surface any unresolved posture
                       concerns from Gates 1-2.

Gate 4 (Engineer):     Implementation must preserve security posture at
                       every system interaction. If an integration requires
                       a posture reduction, STOP and escalate — do not
                       proceed.

Gate 5 (Code           Any code that reduces security posture of data
Reviewer):             compared to its source is a REQUIRED change.

Gate 6 (Quality        Same enforcement as Gate 5. Verify no posture
Engineer):             reduction in any system interaction.

Gate 7 (Security       Posture reduction in any data flow is a CRITICAL
Auditor):              finding. Verify all system interactions preserve
                       posture end-to-end.
```
