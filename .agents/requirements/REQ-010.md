# REQ-10: Coding Standards

## Requirement

**All code produced or modified in this project must follow these language-agnostic coding standards.** These rules are enforced at Gates 4, 5, and 6. Violations are REQUIRED findings that block gate advancement.

Language-specific coding standards are maintained in `.agents/requirements/coding/` and are loaded by the Engineer (Gate 4) based on the languages used in the implementation. Language-specific rules supplement these standards; they do not override them.

---

## Standards

### CS-1: All Imports at Top of File — No Lazy Imports

**All imports, includes, requires, and use statements must appear at the top of the file.** No conditional imports, no imports inside functions, no imports inside branches or loops, no deferred loading of modules.

```
Why top-of-file imports?
----------------------------------------------------------------------
Circular               Lazy imports hide circular dependencies by
dependencies:          deferring the import to a point where the
                       cycle happens not to trigger. This masks a
                       real architectural problem. Top-of-file
                       imports cause circular dependencies to fail
                       immediately and visibly, forcing resolution
                       before the code is merged.

Readability:           A developer scanning a file can see every
                       dependency in the first few lines. No
                       surprises buried in function bodies.

Determinism:           Import side effects (module initialization,
                       registration) happen in a predictable order
                       at startup, not at arbitrary runtime points.
----------------------------------------------------------------------
```

**Exceptions:** None. If a circular dependency prevents top-of-file imports, the architecture must be fixed (extract shared types, introduce an interface, restructure modules). The import is not the problem; the dependency graph is.

### CS-2: Keyword Arguments Over Positional Arguments

**Use keyword arguments (named parameters) wherever the language supports them.** Positional arguments are permitted only where the language requires them (e.g., the first positional argument in some frameworks) or where the meaning is unambiguous from a single argument (e.g., `len(items)`, `str(value)`).

```
Why keyword arguments?
----------------------------------------------------------------------
Readability:           `create_user(name="alice", role="admin")` is
                       self-documenting. `create_user("alice", "admin")`
                       requires checking the function signature.

Refactoring safety:    Reordering or inserting parameters in a function
                       signature silently breaks all positional callers.
                       Keyword callers are unaffected.

Bug prevention:        Swapped positional arguments of the same type
                       (e.g., two strings) produce no type error but
                       cause incorrect behavior. Keywords make this
                       impossible.
----------------------------------------------------------------------
```

**Guidance for ambiguous cases:** If a function call has more than one argument, use keywords. If a single argument's meaning is not immediately obvious from context, use a keyword.

### CS-3: Explicit File Permissions

**When code creates or modifies files programmatically, the file permissions must be set explicitly.** Do not rely on the process umask or OS defaults. The intended permission must be visible in the code.

```
Why explicit permissions?
----------------------------------------------------------------------
Security:              Default permissions vary by OS, user, and umask
                       configuration. A file created with default
                       permissions on a developer machine may be
                       world-readable. Explicit permissions remove
                       this variability.

Auditability:          A reviewer can verify the intended permission
                       by reading the code. If the permission is
                       implicit, the reviewer must know the runtime
                       environment to assess security.

Least privilege:       Explicit permissions force the developer to
                       think about who needs access, rather than
                       accepting whatever the OS provides.
----------------------------------------------------------------------
```

**Minimum expectations:**
- Files containing secrets or credentials: `0600` (owner read/write only)
- Configuration files: `0644` (owner read/write, others read)
- Executable scripts: `0755` (owner read/write/execute, others read/execute)
- Temporary files: `0600` (owner read/write only)

### CS-4: Code Element Ordering

**Code elements must appear in a canonical order within each file.** This standard ensures consistent file structure across the project, making files predictable to navigate and review.

```
Canonical order (top to bottom)
----------------------------------------------------------------------
1. Imports / includes / requires / use statements
2. Constants and module-level configuration
3. Helper functions and utility functions
4. Fixtures (test files only — e.g., pytest fixtures, test setup)
5. Classes (with methods defined inside)
6. Module-level executable code (e.g., if __name__ == "__main__")
----------------------------------------------------------------------
```

```
Why canonical ordering?
----------------------------------------------------------------------
Navigability:      A developer opening any file in the project
                   knows where to look for constants (top, after
                   imports), where to find helpers (middle), and
                   where classes begin (lower). No hunting.

Review efficiency: Code reviewers can verify completeness by
                   scanning top-to-bottom. Constants grouped
                   together are easier to audit for magic numbers.
                   Functions before classes means dependencies
                   are defined before consumers.

Merge conflicts:   Consistent ordering reduces merge conflicts
                   caused by developers placing new elements in
                   different locations within the same file.
----------------------------------------------------------------------
```

**Enforcement tiers:**
- **New files:** Strict. The canonical order is required.
- **Existing files being modified:** Recommended. When modifying an existing file, follow the canonical order for new elements. Re-ordering existing elements to conform is encouraged but not required for unrelated changes.
- **Language or developer overrides:** The canonical order may be overridden when a language convention or framework requirement conflicts with it. The override must be justified (e.g., "Go requires init() to appear after var blocks" or "this framework expects decorators to be co-located with their targets"). Unjustified deviations are findings.

**Exceptions:** Language-specific ordering rules in `.agents/requirements/coding/` may adjust the canonical order for that language. When a language file exists and defines a different order, the language-specific order takes precedence.

### CS-5: De-Duplication of Shared Logic

**When two or more occurrences of similar logic patterns exist across or within files, the duplicated logic must be extracted into a shared location.** Duplication is the root of divergence bugs — when the same logic exists in two places, one will eventually be updated and the other will not.

```
Why de-duplication?
----------------------------------------------------------------------
Consistency:       A single source of truth for each piece of
                   logic guarantees consistent behavior. When a
                   bug is fixed or behavior is changed, it is
                   fixed or changed everywhere.

Maintenance cost:  Every duplicate is a maintenance liability.
                   The cost of updating N copies scales linearly
                   and the probability of missing one scales
                   with it.

Review burden:     Reviewers who see the same logic block in
                   multiple files must verify each copy is
                   identical. Extraction eliminates this burden
                   entirely.
----------------------------------------------------------------------
```

**Threshold:** 2+ occurrences of logic blocks with similar patterns triggers de-duplication. This is intentionally aggressive — catching duplication early prevents divergence.

**What constitutes "similar patterns":**
- Identical or near-identical code blocks (same logic, different variable names)
- Blocks that differ only in configuration values (parameterize instead)
- Repeated setup/teardown patterns across test files (extract to shared fixtures)
- Duplicated validation, transformation, or formatting logic

**Reasoning requirement:** When an agent proposes de-duplication during implementation or review, it must provide clear reasoning to the user:
1. **What** is duplicated — identify the specific blocks and their locations
2. **Why** extraction is warranted — explain the maintenance or correctness risk
3. **Where** the shared logic will live — name the target file or module

No silent refactoring. The user must understand and agree with the extraction before it happens.

**Exceptions:** Duplication that exists for genuine isolation (e.g., two modules that must remain independently deployable with no shared dependencies) is acceptable when the isolation requirement is documented.

## Language-Specific Extensions

Language-specific coding standards are maintained in separate files under `.agents/requirements/coding/`. Each file covers rules that are meaningful only in that language (e.g., Python-specific import ordering, Go-specific error handling patterns).

The Engineer (Gate 4) loads the relevant language file(s) based on the implementation. If no language file exists for a language in use, only the standards in this file apply.

```
.agents/requirements/coding/
  python.md           (future)
  go.md               (future)
  typescript.md       (future)
```

```

                       Load language-specific files from
                       .agents/requirements/coding/ if they exist
                       for the languages in use. Note compliance
                       in the Implementation Report.

Gate 5 (Code           Any CS-* violation is a REQUIRED change.
Reviewer):             Check imports (CS-1), function call style
                       (CS-2), file creation/modification patterns
                       (CS-3), element ordering (CS-4), and
                       duplication (CS-5).

Gate 6 (Quality        Any CS-* violation not caught at Gate 5
Engineer):             is a REQUIRED change.

Note on CS-4:          New files — REQUIRED. Existing files —
                       SUGGESTED (recommended, not blocking).
                       Language/developer overrides — accepted
                       when justified.

Note on CS-5:          De-duplication proposals must include
                       reasoning (what, why, where). Proposals
                       without reasoning are incomplete findings.
```
