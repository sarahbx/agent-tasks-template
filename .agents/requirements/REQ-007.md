# REQ-7: Use .test TLD in All Testing Contexts

## Requirement

**All domain names used in testing contexts must use the `.test` top-level domain (TLD).** This includes but is not limited to: hostnames, URLs, email addresses, API endpoints, DNS names, and any other domain name references in test fixtures, test configuration, test data, and documentation examples. No real or active TLD (`.com`, `.org`, `.net`, `.io`, etc.) may be used in any testing context.

## Rationale

```
Why mandate the .test TLD?
----------------------------------------------------------------------
DNS safety:       The .test TLD is reserved by RFC 2606 ("Reserved Top
                  Level DNS Names," June 1999) and formalized by RFC 6761
                  ("Special-Use Domain Names," February 2013). Names
                  under .test are guaranteed never to be registered in the
                  global DNS. Using .test eliminates any risk of
                  inadvertent DNS resolution to active systems.

Isolation:        Tests that reference real TLDs can inadvertently connect
                  to production systems, leak data, trigger rate limits, or
                  cause unintended side effects on active infrastructure.
                  The .test TLD provides complete isolation by design.

Standards-based:  RFC 2606 and RFC 6761 are IETF standards specifically
                  designed for this purpose. Using .test is not a project
                  convention -- it is adherence to an internet standard.

Predictability:   Engineers do not need to guess which domains are "safe"
                  for testing. The rule is simple and universal: use .test.
----------------------------------------------------------------------
```

## RFC References

```
RFC 2606 - Reserved Top Level DNS Names (June 1999)
----------------------------------------------------------------------
Reserves four TLDs for testing and documentation purposes:

  .test       For use in testing of current or new DNS related code
  .example    For use in documentation or as examples
  .invalid    For use in online construction of domain names that
              are sure to be invalid
  .localhost  For use as the domain name indicating the local host

Additionally reserves second-level domains:
  example.com, example.net, example.org

Reference: https://www.rfc-editor.org/rfc/rfc2606
----------------------------------------------------------------------

RFC 6761 - Special-Use Domain Names (February 2013)
----------------------------------------------------------------------
Formalizes the special-use domain name registry and confirms the
reservations from RFC 2606. Establishes the IANA Special-Use Domain
Names registry for future reservations.

Section 6.2 specifically addresses .test:
  "The domain 'test.' and any names falling within '.test.' are
  special in the following ways..."

  - Users and application software may use test names as they would
    other domain names, but should be aware that they will not resolve
    via the global DNS
  - DNS servers should be aware that test names will not be delegated
    and should not attempt to look them up in the DNS context
  - DNS registries/registrars should not accept registrations of
    test names

Reference: https://www.rfc-editor.org/rfc/rfc6761
----------------------------------------------------------------------
```

## Scope

- **Applies to:** All test fixtures, test configuration, test data, documentation examples, CI/CD pipeline configuration, and any other context where domain names are used for testing purposes
- **Domain name contexts include:**
  - Hostnames and URLs: `api.myservice.test`, `https://app.myservice.test`
  - Email addresses: `user@example.test`, `admin@company.test`
  - API endpoints: `https://api.partner.test/v1/resource`
  - DNS names in configuration: `db.internal.test`, `cache.cluster.test`
  - Certificate subjects and SANs in test certificates
  - Any other reference to a domain name in a testing context
- **Excludes:** References to real domains in documentation that describe external integrations (e.g., "this system connects to the Jira API at `your-instance.atlassian.net`") -- these are descriptions of real systems, not test fixtures
- **Other RFC 2606 reserved TLDs:** While `.example`, `.invalid`, and `.localhost` are also reserved by RFC 2606, this project standardizes on `.test` for consistency. Use `.test` unless a specific RFC 2606 TLD is semantically more appropriate (e.g., `.invalid` for testing invalid domain handling, `.localhost` for loopback testing)
